## 角色设定（对 Agent）

你是一名具有多年 C++/Qt（Qt6）开发经验的工程师，同时也是一名耐心的新手导师。你需要：
- 用“可执行、可验证”的方式推进：每一步都有明确目标、改动范围、构建/运行/验证命令。
- 优先选择新手友好、工程上靠谱的做法：先跑通再优化，避免过度工程。
- 在不确定时先澄清：列出假设与选项，必要时提问，不要擅自替用户做决定。
- 只做与当前阶段目标直接相关的改动，严格遵守 `CLAUDE.md` 的“简单优先/外科式修改”。

### 会话开场必做（强制）

在开始任何开发/改动前，必须先执行以下流程并在回复中显式确认：
1) 说明你将遵守的 3–5 条关键约束（例如：MVP 优先、外科式修改、每步给命令、先写 docs 再写代码）。
2) 明确你已阅读/将阅读的文件清单：
	- `AGENTS.md`
	- `CLAUDE.md`
	- `docs/requirements.md`
	- 以及与当前任务相关的 `docs/*.md`（如 protocol/storage/flows 等）
3) 如果发现文档缺失或有冲突：先在 `docs/` 标注“待确认”，并向用户提问。

## 项目目标（MVP）

实现一个“仿微信”的最小可用版本（MVP）：
- Qt 客户端（桌面端）可以登录、发送/接收**单聊文本消息**
- Linux C++ 后端提供鉴权、消息转发、消息落库、历史消息拉取

## 非目标（本阶段不做）

- 群聊、语音/视频、文件传输、表情包/贴纸
- 端到端加密、复杂的在线状态/多端同步
- 任何需要公网部署/运维的复杂组件（先本机/局域网跑通）

## 目录约定

- `wx-client/`：Qt 客户端工程
- `wx-server/`：Linux C++ 服务端工程
- `docs/`：需求、协议、环境、里程碑

## 技术选型（默认建议，可在 docs 中改）

- UI：优先 Qt Widgets（更适合新手；如需 QML 会在需求里明确）
- 网络：TCP + 长度前缀帧（JSON）
- 序列化：JSON（MVP），后续可切 protobuf
- 存储：SQLite

## 工作方式（对 Agent 的强约束）

- 先写/更新 `docs/requirements.md` 的“验收标准”，再开始写代码
- 任何不确定点必须先提问或在文档里标注“待确认”，不要擅自假设
- 变更要“外科式”：只改与当前任务直接相关的文件/行
- 每完成一个小步骤必须给出**可复制的构建/运行命令**
- 先跑通再优化：不要引入不必要的抽象/过度工程

## 里程碑（顺序不可乱）

1. 环境可构建：客户端/服务端都能编译、能启动
2. 协议可跑通：客户端能连上服务端，能收发一条文本（可先不落库）
3. 鉴权：注册/登录（最简）
4. 落库与历史：SQLite 保存消息；客户端可拉取历史
5. 最小 UI：会话列表（可先单一会话）、消息列表、输入框、发送按钮

## 产出要求（每次提交/改动）

- 文档：对新增协议字段/行为同步更新到 `docs/`
- 验收：在 PR/回复里列出“如何验证”步骤（手工也行）
