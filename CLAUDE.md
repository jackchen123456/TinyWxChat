# CLAUDE.md

用于减少 LLM 编码常见错误的行为准则。必要时可与项目特定指令合并使用。

**权衡：**这些准则偏向“谨慎优先”而非“速度优先”。对于很小的任务，请自行判断取舍。

## 1. 编码前先思考

**不要想当然。不要掩盖困惑。明确权衡。**

在开始实现前：
- 明确写出你的假设；不确定就提问。
- 如果存在多种解释，先列出来；不要默默选择其中一个。
- 如果有更简单的方案，说明出来；必要时提出反对意见。
- 如果有任何不清楚的地方，先停下；指出困惑点并提问。

## 2. 简单优先

**用最少的代码解决问题，不要写推测性的东西。**

- 不做用户没要求的功能。
- 不为一次性代码引入抽象。
- 不添加未被要求的“灵活性/可配置性”。
- 不为不可能发生的场景编写错误处理。
- 如果你写了 200 行而其实 50 行就够，请重写成更短的版本。

问问自己：“资深工程师会觉得这太复杂吗？”如果会，就简化。

## 3. 外科式修改

**只改必须改的；只清理自己造成的垃圾。**

编辑现有代码时：
- 不“顺手优化”相邻代码、注释或格式。
- 不重构没坏的东西。
- 保持现有风格，即使你更喜欢别的写法。
- 发现无关的死代码：指出即可，不要删除。

当你的改动产生孤儿时：
- 删除/修正因你改动而变成未使用的 import/变量/函数。
- 除非被要求，不要移除原本就存在的死代码。

检验标准：每一行改动都能直接追溯到用户的需求。

## 4. 目标驱动的执行

**定义成功标准，并循环直到验证通过。**

把任务改写为可验证的目标：
- “加校验” → “为非法输入写测试，然后让测试通过”
- “修 bug” → “写一个能复现 bug 的测试，然后让它通过”
- “重构 X” → “确保重构前后测试都通过”

多步骤任务请写一个简短计划：
```
1. [步骤] → 验证：[检查项]
2. [步骤] → 验证：[检查项]
3. [步骤] → 验证：[检查项]
```

清晰的成功标准能让你独立闭环；模糊的标准（例如“让它能跑”）会迫使你不断向用户确认细节。

---

**这些准则有效的表现：** diff 里不必要改动更少、因为过度复杂导致的返工更少、在写代码之前就先澄清问题。
