# 需求与验收（分阶段）

> 目标：实现一个功能较完善的“仿微信”项目，同时保持可分阶段交付、每阶段可验收。

## 0. 已确认的前提（决策记录）

- 开发环境：Ubuntu
- 客户端 UI：Qt Widgets（Qt6）
- 服务端：支持**多客户端同时在线**（多连接并发管理）
- 运行形态：优先本机/局域网跑通（后续再考虑公网部署）
- 协议：TCP + 长度前缀帧 + JSON（先跑通；后续可替换为 protobuf）
- 登录方式：用户名 + 密码（不做短信/验证码）

## 1. 角色与核心场景

### 用户角色
- 普通用户：注册/登录、单聊与群聊、收发文本/表情/图片、查看历史、处理好友/群聊申请

### 核心场景（从易到难）
1) 注册/登录
2) 单聊文本：A 发给 B，B 在线即时收到
3) 单聊离线：B 离线，B 下次登录能看到历史/未读
4) 会话列表：展示最近会话并切换
5) 表情与图片：发送/展示，图片可预览
6) 群聊：创建/加入、群消息广播
7) 申请流：好友申请、群聊申请（最简：申请/同意）

## 2. 功能范围（最终形态）

### 客户端（Qt Widgets）
- 登录页面
- 注册页面
- 会话列表（会话切换）
- 用户聊天页面（单聊）
- 群聊页面
- 好友申请（查看/同意）
- 群聊申请（查看/同意）
- 选择头像
- 发送表情
- 图片发送
- 图片预览

### 服务端（Linux C++）
- 账户：注册/登录
- 连接与在线：多客户端同时在线；同一用户多端登录策略（见“未决问题”）
- 单聊：消息转发 + 落库 + 历史拉取
- 群聊：群管理（最简）+ 群消息广播 + 落库 + 历史拉取
- 申请流：好友申请、群聊申请（最简：申请/同意）

## 3. 分阶段路线图（建议）

### Phase 0：工程可构建（基础设施）
- 客户端/服务端工程骨架 + CMake
- 本机能编译、能启动、能打印日志
-（文档）补齐：`docs/protocol.md`、`docs/storage.md`、`docs/flows.md`

### Phase 1：MVP（聊天闭环）
- 登录
- 单聊文本收发（至少 2 个客户端同时在线可互发）
- SQLite 落库 + 拉取最近 N 条历史（默认 N=20）

### Phase 2：接近微信的基础体验
- 注册
- 会话列表（至少 2 个会话切换）
- 好友申请（申请/同意）
- 头像选择与展示

### Phase 3：富媒体与群聊
- 表情发送与展示
- 图片发送与预览（先限制大小/类型，后续再放开）
- 群聊页面（创建/加入/消息广播）
- 群聊申请（申请/同意）

## 4. 验收标准（按阶段）

### Phase 0 验收
- 服务端可启动：监听端口且无崩溃
- 客户端可启动：能打开窗口且无崩溃
- 文档存在且一致：`docs/protocol.md`、`docs/storage.md`、`docs/flows.md`

### Phase 1 验收（MVP 必须全部满足）
- 登录成功后客户端与服务端保持连接
- 单聊文本：A→B 发送成功，B 实时收到（至少 2 个客户端实例）
- 历史消息：发送过的消息写入 SQLite；登录后能拉取并展示最近 20 条

### Phase 2 验收
- 注册成功后可登录
- 会话列表展示最近会话，能切换会话并正确展示消息
- 好友申请：A 发起申请，B 看到并同意；同意后允许发起会话
- 头像：可选择头像并在 UI 中展示

### Phase 3 验收
- 表情：能发送并在对端正常展示
- 图片：能发送；对端可点击预览
- 群聊：创建/加入后，群消息能广播到在线成员；群消息落库并可拉历史
- 群聊申请：申请/同意流程可用

## 5. 非功能需求（最小但必须）

- 稳定性：服务端可同时处理多个客户端连接，不应因单个客户端异常而崩溃
- 可观测性：客户端/服务端都有基本日志（连接、登录、发消息、错误）
- 可调试性：给出明确的构建/运行/验证命令（每个阶段都要）

## 6. 未决问题（实现前需定稿，避免返工）

1) 同一用户多端登录：允许/不允许？允许则消息如何同步到多个连接？
2) 图片存储：先存本地文件路径还是二进制/对象存储？（建议 MVP 先存本地目录 + 记录元数据）
3) 表情：先用内置表情 ID 映射（推荐）还是自定义图片？
