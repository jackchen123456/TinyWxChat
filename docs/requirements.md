# 需求与验收（分阶段）

> 目标：实现一个功能较完善的"仿微信"项目，同时保持可分阶段交付、每阶段可验收。
> 最后更新：2026-04-28（v0.2 — 协议草案后修订）

## 0. 已确认的前提（决策记录）

- 开发环境：Ubuntu
- 客户端 UI：Qt Widgets（Qt6）
- 服务端：支持**多客户端同时在线**（多连接并发管理）
- 运行形态：优先本机/局域网跑通（后续再考虑公网部署）
- 协议：TCP + 二进制帧头 8B + JSON Payload（先跑通；后续可替换为 protobuf，帧头兼容）
- 登录方式：用户名 + 密码（不做短信/验证码）
- 存储引擎：SQLite 3（WAL 模式），服务端本地文件 `wx-server/data/tinywechat.db`

### 0.1 已定稿的决策（原"未决问题"）

1) **同一用户多端登录**：不允许。新登录会踢掉旧连接（服务端在 login 成功时主动断开该 user_id 的旧连接）。
2) **图片/文件存储**：存服务端本地目录（如 `wx-server/data/avatars/`），数据库仅记录相对路径 + 元数据。MVP 跳过，Phase 3 再实现。
3) **表情**：使用内置表情 ID 映射（客户端内置表情集，协议传 ID）。MVP 跳过，Phase 3 实现。
4) **图片传输**：Phase 3 采用客户端内嵌 base64 方式——客户端将图片 base64 编码后放入 `content` 字段，`msg_type=3`，服务端原样转发和存储。单张图片原始大小最大 500KB（base64 后约 670KB）。客户端发送前应压缩/缩放至合理尺寸。后续可升级为"文件服务 + URL 引用"模式。
5) **会话列表**：采用从 `messages` 表聚合的方案（不建独立 conversations 表），通过 `SELECT DISTINCT ... GROUP BY peer_id ORDER BY last_time DESC` 查询最近会话。MVP 阶段不维护未读计数。
6) **删除好友**：当前版本不实现。好友关系只增不减，用户如需"屏蔽"可通过不发起会话实现。后续版本可考虑。

### 0.2 关键约束（对开发 Agent）

- MVP 优先：Phase 1 只做登录 + 单聊文本 + 落库/历史，不做注册/群聊/好友/表情
- 外科式修改：只改与当前任务直接相关的文件
- 先跑通再优化：每步给构建/运行/验证命令
- 协议更新必须同步到 `docs/protocol.md`
- 聊天消息包含 `msg_type` 字段（目前仅值为 1），为 Phase 3 富媒体预留

## 1. 角色与核心场景

### 用户角色
- 普通用户：注册/登录、单聊与群聊、收发文本/表情/图片、查看历史、处理好友/群聊申请

### 核心场景（从易到难）
1) 注册/登录
2) 单聊文本：A 发给 B，B 在线即时收到
3) 单聊离线：B 离线，B 下次登录能看到历史/未读
4) 会话列表：展示最近会话并切换
5) 表情与图片：发送/展示，图片可预览
6) 群聊：创建/加入、群消息广播
7) 申请流：好友申请、群聊申请（最简：申请/同意）

## 2. 功能范围（最终形态）

### 客户端（Qt Widgets）
- 登录页面
- 注册页面
- 会话列表（会话切换）
- 用户聊天页面（单聊）
- 群聊页面
- 好友申请（查看/同意）
- 群聊申请（查看/同意）
- 选择头像
- 发送表情
- 图片发送
- 图片预览

### 服务端（Linux C++）
- 账户：注册/登录
- 连接与在线：多客户端同时在线；同一用户不允许多端登录（新登踢旧）
- 单聊：消息转发 + 落库 + 历史拉取
- 群聊：群管理（最简）+ 群消息广播 + 落库 + 历史拉取
- 申请流：好友申请、群聊申请（最简：申请/同意）
- 映射维护：`session_id → ConnectionState` + `user_id → session_id`
- 错误码：统一整型错误码体系（见 `docs/protocol.md` §4）

## 3. 分阶段路线图（建议）

### Phase 0：工程可构建（基础设施）
- 客户端/服务端工程骨架 + CMake
- 本机能编译、能启动、能打印日志
- **TCP 连接就绪**：客户端能 TCP connect → 服务端 accept，完成一次 PING/PONG
-（文档）补齐并交叉一致：`docs/protocol.md`、`docs/storage.md`、`docs/flows.md`

### Phase 1：MVP（聊天闭环）
- 登录（用户名+密码，bcrypt 验证）
- 单聊文本收发（至少 2 个客户端同时在线可互发，msg_type=1）
- SQLite 落库（messages 表）+ 拉取最近 N 条历史（默认 N=20）

### Phase 2：接近微信的基础体验
- 注册
- 会话列表（至少 2 个会话切换）
- 好友申请（申请/同意）
- 头像选择与展示

### Phase 3：富媒体与群聊
- 表情发送与展示（内置 ID 映射）
- 图片发送与预览（先限制大小/类型，后续再放开）
- 群聊页面（创建/加入/消息广播）
- 群聊申请（申请/同意）

## 4. 验收标准（按阶段）

### Phase 0 验收
- 服务端可启动：监听指定端口且无崩溃，打印启动日志
- 客户端可启动：能打开窗口且无崩溃，打印启动日志
- **TCP 连接验证**：客户端能 TCP connect → 服务端 accept → 完成一次 PING/PONG
- 文档存在且交叉一致：`docs/protocol.md`、`docs/storage.md`、`docs/flows.md`
- 构建命令可复制执行：`mkdir build && cd build && cmake .. && make`

### Phase 1 验收（MVP 必须全部满足）
- 登录成功后客户端与服务端保持连接
- 单聊文本：A→B 发送成功，B 实时收到（至少 2 个客户端实例）
- 历史消息：发送过的消息写入 SQLite（messages 表）；登录后能拉取并展示最近 20 条
- 心跳正常：30s PING / 90s 超时断开

### Phase 2 验收
- 注册成功后可登录
- 注册边界校验：
  - 用户名为空 → code=200 (INVALID_REQUEST)
  - 用户名超过 32 字节 → code=402 (USERNAME_TOO_LONG)
  - 密码少于 6 位 → code=200 (INVALID_REQUEST)
  - 用户名重复 → code=400 (DUPLICATE_USERNAME)
- 会话列表展示最近会话，能切换会话并正确展示消息
- 会话列表无会话时显示"暂无会话"空状态提示
- 好友申请：
  - A 发起申请，B 在线时实时收到 friend.notify 通知
  - B 不在线时，下次登录可通过 friend.pending 查询到
  - B 同意后双方 friend.list 均可见对方
  - 不能向自己发申请 → code=200
  - 已是好友再申请 → code=311 (FRIEND_ALREADY)
  - 重复申请（对方未处理）→ code=312 (FRIEND_REQUEST_EXISTS)
- 头像：可在本地资源中选择头像并在 UI 中展示

### Phase 3 验收
- 表情：
  - 点击表情按钮弹出 EmojiPicker 面板
  - 选择表情后插入到输入框
  - 发送后在对端正常展示（Unicode 表情字符）
- 图片：
  - 点击附件按钮打开文件选择器（png/jpg/gif/bmp）
  - 选择后以 msg_type=3 发送（content 含 base64 编码图片数据）
  - 对端收到后在消息列表中展示图片预览
  - 单张图片原始大小超过 500KB 时应提示用户
- 群聊：
  - 创建群聊：输入群名 → 返回 group_id
  - 加入群聊：输入 group_id → 加入成功
  - 群消息广播：成员发送后，在线成员实时收到 group.recv
  - 群消息落库：离线成员下次拉群历史可见
  - 群历史拉取：默认最近 20 条
  - 非群成员发送消息 → code=322 (GROUP_PERMISSION_DENIED)
  - 重复加入已加入的群 → code=321 (GROUP_ALREADY_MEMBER)
- 群聊申请：
  - 申请加入群聊（群主可审批）
  - 群主同意后申请人成为群成员

## 5. 非功能需求（最小但必须）

- 稳定性：服务端可同时处理多个客户端连接，不应因单个客户端异常而崩溃
- 可观测性：客户端/服务端都有基本日志（连接、登录、发消息、每个消息的 type+seq、错误码）
- 可调试性：给出明确的构建/运行/验证命令（每个阶段都要）
- 安全性：密码使用 bcrypt 哈希存储，不存明文

|------|------|
| `docs/protocol.md` | 帧格式、消息类型(25+)、body 字段、错误码(0~501)、心跳、连接生命周期 |

## 7. 已知限制与后续优化（非本阶段）

- 消息查询 OR 双支索引优化（`idx_messages_rev`）
- 协议切 protobuf
- 消息自动清理（超过 N 天）
- 连接数上限与压力测试
- Session 对象在客户端断开后内存不释放（MVP 可接受，后续用 shared_ptr 管理）
- 图片仅支持客户端本地路径（无法跨机器，后续升级为文件服务）
- 不支持消息删除/撤回
- 不支持用户信息修改（昵称、密码）
- 不支持删除好友（当前只增不减）

## 8. 客户端 UI 规范（最低要求）

### 窗口与导航
- 登录窗口标题：`TinyWeChat 登录`
- 登录成功后跳转到主界面（标签页导航：聊天 | 好友 | 群聊）
- 聊天页面顶部显示对方昵称，左侧有 `← 返回` 按钮回到主界面

### 消息展示
- 自己发的消息：蓝色气泡 `#d6eaf8`，右对齐
- 收到的消息：绿色气泡 `#d5f5e3`，左对齐
- 系统消息（如历史拉取提示）：居中灰色显示
- 每条消息显示发送者昵称和时间戳（HH:mm 格式）

### 输入与交互
- 消息输入框支持回车键发送
- 消息内容最大 4096 字节，超长时应给出提示
- 表情按钮（😊）在输入框左侧，点击弹出表情面板
- 图片按钮（📎）在输入框左侧，点击打开文件选择器

### 错误与状态
- 连接失败时显示红色错误提示
- 登录失败时显示具体错误原因（如"用户名或密码错误"）
- 发送消息失败时在消息列表中显示红色系统提示
